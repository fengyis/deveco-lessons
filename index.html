长程任务案例实战演练：
让 Agent 按计划稳定完成复杂开发任务

一天做三件事：看清 Agent 做了什么，让它按计划继续，并用独立证据决定何时收工。

OPENING · LOOP STACK

Loop Engineering 的四层循环

长任务常卡在四处：没做完、做错了、停住了、同类错误反复出现。每层 Loop 处理一处。

框架来源：LangChain · The Art of Loop Engineering

LOOP 1 · AGENT把任务做完
LOOP 2 · VERIFY把任务做对
LOOP 3 · EVENT停了自己续
LOOP 4 · IMPROVE跑完留经验
  1. LOOP 1

    Agent Loop

    自动完成工作

    怎么跑：读取上下文 → 调工具 → 写回结果 → 继续。单独使用适合边界明确、可人工抽查的小任务。

  2. LOOP 2

    Verification Loop

    保证质量与正确性

    怎么跑：测试或 Grader 判分；未过线就把差距退回重做。适合迁移、重构和 CI 修复。

  3. LOOP 3

    Event-driven Loop

    接入业务系统持续运行

    怎么跑：消息、Webhook、Cron 或 Session 事件触发下一轮。适合工单、巡检和长流程续推。

  4. LOOP 4

    Hill-climbing Loop

    用运行数据改进 Harness

    怎么跑：从 Trace 找共性失败，改 Prompt、Tool、Grader 或 Skill，再回归评测。适合高频重复任务。

不用一次上齐四层。先让任务跑通；需要质量保证时加验证；需要长期运行时接事件；积累足够 Trace 后再优化 Harness。高风险动作始终留给人工审批。

快速开始

唯一前置：已安装 deveco（npm install -g @deveco/deveco-code）。 其余依赖——包括 vendor 在 vendor/ 下的 cannbot-insight 本体——全部由 setup.sh 就地装好。

~/deveco-lessons
$ git clone <本仓库> && cd deveco-lessons
$ ./setup.sh            # 一键环境：node20 / bun / cannbot 依赖与数据库，幂等可重跑
$ deveco auth login     # 唯一需要你自己完成的一步：配模型凭证

装完先从观测实操开始：先建立可审计现场，再进入 Loop Engineering。

LESSON 1

建立细粒度审计轨迹：DevEco 对接 cannbot-insight

DevEco 与 cannbot-insight 使用兼容的 session / message / part 数据结构。 observe.sh 直接导入会话库并启动回放界面，无需改 DevEco。

你会学到

  • DevEco 的会话数据长什么样、存在哪
  • 为什么 cannbot-insight 能零改码直读 DevEco 的会话库
  • 用 observe.sh 把任意项目的会话导入并在 Web 界面回看

步骤

  1. 造几条会话 —— 随便建个目录，用 deveco 跑两三个会话（有工具调用更好看）： 
    $ mkdir -p ~/play/hello-deveco && cd ~/play/hello-deveco
$ deveco run "写一个 fizzbuzz.py 并运行它"
$ deveco run "把 fizzbuzz 改成 1 到 30 并重新运行"
  2. 导入观测 —— 回到本仓库，按项目目录捞出所有 root 会话、逐个导入（按 taskId 去重，幂等）： 
    $ cd <本仓库>
$ ./lesson1-insight/observe.sh ~/play/hello-deveco
✅ 观测就绪: http://localhost:21025
  3. 回看 —— 浏览器开 http://localhost:21025，点进会话逐个看分析页： tokens/成本、上下文增长、工具调用、工作流阶段、概念传播。

    重复运行 observe.sh 会按 taskId 更新已有记录，会话数不会翻倍。

验收

  • 能在 Web 界面指出某一轮的输入/输出 token 数
  • 能说出该会话调用了哪些工具、各调了几次
  • 对同一项目重跑 observe.sh，会话数不翻倍（幂等）

坑

  1. node 版本：better-sqlite3 只在 node 20 编得过。脚本会自动切 nvm 的 node 20，手动进 vendor 装依赖要先 nvm use 20。
  2. observe 是旁路：cannbot 没装好、server 起不来、deveco.db 不存在时，脚本会提示原因并以退出码 0 结束，调用方可以继续运行。
  3. server 日志在 /tmp/cannbot-insight.log，起不来先看它。
MODULE 02 · CONTROL

Loop Engineering：让 Agent 按计划稳定完成复杂任务

每轮只做一小步；验证通过就记进度，失败就带着差距重做。

定义目标→ 制定计划→ 执行最小增量→ 独立验证→ 更新状态 / 反馈↺ 证据充分，完成
稳定收敛的四个工程条件
DECOMPOSE

任务可分解

拆成有顺序、可单独验收的小步；每轮只领一项。

GOAL → PLAN → 本轮子任务
VERIFY

每轮可验证

用测试、编译或只读 reviewer 给出客观结果。

本轮产出 → 独立验证器判分 → 通过 / 差距喂回下一轮
STATE

状态可继承

用 Git 和进度文件记住完成项、失败尝试与下一步。

commit + PROGRESS → 下一轮接着干
FEEDBACK

反馈可执行

反馈给出失败样例和根因，能直接变成下一轮任务。

差距（样例/根因）→ 注入下一轮

边界：限制轮次、时长、token 和工具权限；发布、删数据等不可逆动作停在人工确认前。

任务可分解+ 每轮可验证+ 状态可继承+ 反馈可执行⇒ 适合 Loop Engineering

下面看两种实现：/goal 管单会话长目标，Ralph 用独立 reviewer 做逐轮验收。

LESSON 2

devecocode-goal-plugin：给会话装上目标循环

Agent 停下但目标未完成时，session.idle 自动触发下一轮；提交完成证据后才允许退出。 循环留在当前会话内，无需外部 shell。（适配自 opencode-goal-plugin v0.6.5。）

一张图看运行方式

/goal 设目标→ Agent 执行→ session.idle 检查↺ 未完成 证据通过，归档

你会学到

  • 目标如何设定、续推、验收与归档
  • session.idle 如何触发下一轮
  • 轮次、时长和 token 预算如何控制退出

用法

  1. 装 —— goal.sh 只负责装：插件 + merge deveco.json 的 /goal 命令： 
    $ ./goal.sh init ~/my-project     # --update 才会覆盖你改过的文件
  2. 会话里设目标 —— 交互式 TUI 形态，一切在单个会话内发生： 
    $ cd ~/my-project && deveco
# 会话里输入：
/goal 创建 hello.txt 并 git commit
/goal status     # 看当前目标
/goal history    # 看 lifecycle 事件
  3. 观察续推 —— Agent 停下但目标未完成时，插件自动注入下一轮消息。
  4. 检查退出 —— 完成汇报要通过证据审计；状态归档在 .deveco/goals/state.json。 退出条件可配（.deveco/goal-plugin.json：maxTurns 10 / maxDurationMs 900000 / maxTokens 200000）。 排障第一句永远是：看 .deveco/goals/plugin.log 有没有加载记录。

当前进度

  • 上游 v0.6.5 源码与 9 个测试已 vendor，基线全绿（upstream.lock 锁定 commit）
  • 适配任务按 10 步计划推进中：状态路径迁 .deveco/、session API 注入 query.directory、入口壳、goal.sh、冒烟、E2E 走查（见 docs/superpowers/plans/2026-07-16-lesson3-goal-plugin.md）
LESSON 3

Ralph Loop：让 Agent 按计划跨会话推进

Worker 做一小步，reviewer 独立验收；未通过就退回差距，通过才写 DONE。 验收测试与 worker 隔离。

三件套

worker每轮挑一个最小可验证子任务做掉，写进 .ralph/PROGRESS.md，git commit。
reviewer只读裁判，用 git log / git diff 核实，最后一行必须是 DONE 或 CONTINUE: ⟨还差什么⟩。
插件监听 session.idle → 开 reviewer 会话裁决 → DONE 写 .ralph/DONE 收工，否则把 worker 再踢一轮。

步骤（先拿 smoke 案例练手，30 秒验证整条链路）

  1. 装 —— init 是有副作用的一次性动作（插件、agent、git、配置、GOAL 骨架），所以和「跑」分开： 
    $ cd lesson3-ralph-loop
$ ./ralph.sh init ~/ralph-smoke
  2. 灌目标 —— 用自带案例；自己出题则编辑 .ralph/GOAL.md，验收标准必须「能被 git/测试客观核实」： 
    $ ./ralph.sh sample smoke ~/ralph-smoke   # 案例带的独立验收测试一并装进 .ralph/

    验收测试放在 .ralph/verify.test.ts。worker 拥有全部工具权限，能够修改 test/ 下的用例；bun test 不扫描隐藏目录，因此 worker 看不到也改不到这份验收测试。

  3. 跑 —— 起 server、启动循环、盯到收工、自动关 server；脚本会保留你改过的插件和提示词： 
    $ ./ralph.sh run ~/ralph-smoke
# 想看单次基线（无 reviewer、无续轮）：./ralph.sh once ~/ralph-smoke
# 想实时围观：deveco attach http://127.0.0.1:4097 --session $(cat ~/ralph-smoke/.ralph/session_id)
  4. 看结果 —— 结束标志二选一，然后自己复验： 
    .ralph/DONE      # 验收通过
.ralph/STOPPED   # 到 maxIterations（默认 20，首跑建议设 3）仍未通过
$ bun test ./.ralph/verify.test.ts   # 独立验收测试复验
  5. 回看轨迹 —— run 收工后自动调第一课的 observe.sh，把 worker / reviewer 会话导进 cannbot-insight， 开 http://localhost:21025 turn-by-turn 回看每轮的 token 与工具调用。

    关闭自动观测：RALPH_NO_OBSERVE=1 ./ralph.sh run ...；手动补一次：./ralph.sh observe ⟨项目⟩。观测作为旁路运行，异常不会影响 run 的结论。

DevEco 适配的四个要点

  1. 配置目录：插件配置放在 .deveco/。写入 .opencode/ 会静默失效；.ralph/plugin.log 的生成记录可以确认插件已经加载。
  2. 配置文件：项目配置使用 deveco.json 或 deveco.jsonc；opencode.json 不会激活 DevEco 配置。
  3. SDK 参数结构：DevEco SDK 采用 hey-api 风格，调用形式为 session.prompt({ path: { id }, query: { directory }, body: {...} })。扁平参数会导致会话创建失败；启动首轮会话直接调用 HTTP API，跳过当前存在兼容问题的 deveco run --attach。
  4. Server 鉴权：保持 DEVECO_SERVER_PASSWORD 未设置。该变量会为 serve 开启 basic auth，插件内 client 调用同一 server 时会收到 401，循环随即空转。

实验实证：单次 Agent vs Ralph Loop

同一模型、同一任务、同一套隐藏验收，只改变一件事：有没有反馈环。

实验设计

把 CPython textwrap 移植到 Rust；816 条验收向量对 worker 隐藏。 A 组运行一次，B 组接收 reviewer 的失败反馈后继续。

A 组 · ONCE（无反馈环）
795/816 自认完成
wrap 760/780 · fill 13/14 · 其余全对■ 21 条失败
  • 一次写完、自测全绿、收工，全程约 10 分钟、1 个实现 commit
  • 21 条失败全部聚在 break_on_hyphens 断连字符的断点语义——弱模型自测扫不到的长尾
  • 关键观察：它停手时不知道自己错了多少。没人告诉它哪错了，795 就是终点
B 组 · LOOP（reviewer 隐藏验收反馈）

dedent · indent · wrap · fill · shorten 全部满分

  • CONTINUE 精确点出最后一条失败的根因与修法 → worker 用 1 个 commit 修复
  • 第 2 轮复核约 2 分钟 → DONE。多花的约 14 分钟，买到的是确定的正确

围绕这些问题聊
什么任务适合 Loop？点击展开参考答案

能拆分、能验证、能继承进度，且失败反馈能直接指导下一轮。典型如迁移、重构和 CI 修复。

什么任务不适合 Loop？

主观探索、不可逆操作，以及验证成本高于任务本身的小任务。

Loop Engineering 和 Harness 是什么关系？

Harness 是 Prompt、工具、验证器、状态与边界；Loop Engineering 是把它们接成“执行 → 验证 → 反馈”的回路。

如何优化 Loop？

回放失败轨迹找共因，一次只改一个变量，再用同一任务复测分数曲线。

课堂挑战 · 不给测试用例，把通过率提上去

挑战：不泄漏隐藏用例，让小鲁班模型的验收分数继续爬升

重跑 experiment1 的 A/B 双臂，只改 Harness，不让 worker 接触 816 条验收向量。

  • 可改：目标、提示词、轮次与自检策略
  • 红线：worker 不得读取验收文件；审计不 clean，成绩作废
  • 交付：两臂分数、审计结论、改动与解释

MODULE 04 · TRANSFER

代码迁移优秀实践：MigBot 鸿蒙化案例

MigBot 已把 18 款 APP、160 万行安卓代码迁上鸿蒙，工具迁移环节零人工编码。这背后是完整的工程体系：迁移知识库、规则引擎、skill 化的流水线，以及一套按 Loop Engineering 搭起来的执行 harness。本节聚焦其中的 harness 部分——它回答的是"模型不可靠时，怎么让结果可靠"这个问题，核心是三种 loop：Verify & Fix、event-driven 的多 Session Reviewer、熵减对抗的 Generator vs Discriminator。先看效果，再逐个拆开。

02

MigBot 里用了三种 Loop

80+ skills 负责迁移，三种 Loop 分别守住任务、会话和整仓质量。

LOOP 01 · 任务内

Verify & Fix

编译、契约、测试逐项检查；失败带证据重做，超限转人工。

LOOP 02 · 会话间

Event-driven 多 Session Reviewer

Reviewer 挡住不合格产出；上下文接近上限时换会话续跑。

LOOP 03 · 整仓

熵减对抗 Generator vs Discriminator

评估器算偏差 H，生成器逐条修复；只保留 H 下降的版本。

三种循环各守一段，03、04、05 逐个拆开。

03

Verify & Fix 把迁移过程沉淀为门控契约

a2h-verify 顺序执行 11 项检查；只出报告，修复循环由各 verifier skill 负责。

Verify & Fix 是 skill 驱动的 loop：验收标准与修复循环都封装在 skill 里——AI 写代码，skill 门控把关，人审阅关键产物。

04

多 Session Loop：把几小时的长程迁移压回有效工作区间

Reviewer 防跑偏；Session Pool 在上下文过长前换会话。

05

ECAT 熵减对抗：质量从门控走向自动收敛

Verify & Fix 只判通过与否；ECAT 用总熵 H 表示剩余偏差，并逐轮降低。

什么是「熵减对抗」

01 · 度量编译、功能、UI 等偏差加权为总熵 H；H 越低越接近完成。
02 · 分工评估器找问题，生成器按问题清单修复。
03 · 接受H 下降才保留新版本；低于阈值且稳定时停止。

全景

Generator ⇄ Discriminator 对抗回路

评估器 Discriminator

职责
给迁移质量打分、定位偏差
维度
15 维加权评分：编译、静态检查、页面审计、功能一致性、UI 对齐、幻觉检测、运行时存活…
输出
问题清单 + 总熵 H（离「迁移完成」的距离）

生成器 Generator

职责
按问题清单逐条修复偏差
能力来源
调用 core 已有的修复能力，不另造一套迁移技能
停止条件
H 低于阈值且稳定 → 收敛停；或达到最大迭代数

算法：循环写成 17 行伪代码

Evaluate输出剩余偏差 H 和逐条修复指引 g。
GateH < ε 才退出；完成由评估器决定。
Generate按 g 修复仓库，并参考已验证的成功轨迹。
AcceptH 下降才接受新仓，否则回退。

工程护栏

01 · 回归回退相邻轮 H 回升超出容忍度，自动 git checkout 撤销。
02 · 震荡检测连续多轮无实质进展，提前终止不空烧。
03 · 每轮验证每轮结束验证编译；循环中保留人工介入通道。

这 17 行只做一件事：让 H 单调降到 ε——D 评估出 (H, g)，G 按 g 修复，H 降了才接受新仓并把轨迹蒸馏进记忆树 M，没降就整轮作废回退。修复能力来自 core 的 skill，ECAT 的增量是这条收敛回路。

MODULE 05 · CLOSE

总结与交流

最后只检查四件事：看得见，按计划，做得完，有证据。

  1. 看得见Trace 能区分推进、空转、跑偏和验收污染。
  2. 按计划任务可分解，每轮可验证，状态可继承，反馈可执行。
  3. 做得完/goal 管长目标；Ralph 在关键节点逐轮验收。
  4. 有证据代码、测试、diff、日志和独立裁决共同支持 DONE。
