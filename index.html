长程任务案例实战演练：
让 Agent 按计划稳定完成复杂开发任务

Loop Engineering 以适配任务的 Harness 为工程边界，叠加执行、验证、事件触发与持续改进四层循环，让 Agent 稳定完成长程任务。

OPENING · LOOP STACK

Loop Engineering 的四层循环

可靠 Agent 由模型、工具和适配任务的 Harness 共同组成。四层 Loop 从完成一次任务逐级扩展到持续运行和持续改进，每一层都包裹并增强前一层。

框架来源：LangChain · The Art of Loop Engineering

LOOP 1 · AGENT做完任务
LOOP 2 · VERIFY做对任务
LOOP 3 · EVENT持续运行
LOOP 4 · IMPROVE持续变好
  1. LOOP 1

    Agent Loop

    自动完成工作

    循环：模型读取上下文，选择工具并执行，再把结果写回上下文，直到完成条件满足。

    适合：边界明确、工具齐备、单次交付可人工抽查的任务，例如资料检索、小功能修改和一次性脚本。

    本课落点：DevEco worker 与 /goal 中的执行内循环。

  2. LOOP 2

    Verification Loop

    保证质量与正确性

    循环：Grader 按测试、规则或 Rubric 评分；未通过时把具体差距反馈给 Agent 重试。每轮验证都会增加延迟和成本。

    适合：正确性和一致性优先的生产任务，例如代码迁移、重构、CI 修复、契约和合规检查。

    本课落点：Ralph reviewer、隐藏验收测试与 Verify & Fix。

  3. LOOP 3

    Event-driven Loop

    接入业务系统持续运行

    循环：消息、Webhook、Cron 或 Session 事件触发 Agent，执行结果再写回业务系统。

    适合：后台持续运行、异步到达、定时或批量任务，例如工单处理、仓库维护、监控响应和长流程续推。

    本课演示：session.idle 与 /goal 展示事件触发的单会话续推；完整业务接入还需要 Webhook、Cron 或 Channel 触发器。

  4. LOOP 4

    Hill-climbing Loop

    用运行数据改进 Harness

    循环：分析生产 Trace 和 Grader 反馈，识别共性失败，更新 Prompt、Tool、Grader、Skill 或规则，再发布新的 Harness。

    适合：同类任务高频重复、评价标准稳定、已经积累足够 Trace 的场景。

    本课落点：cannbot-insight 提供 Trace 输入；分享点评、经验写回与 Harness 发布组成由人工参与的改进闭环。

选择原则：先建立能工作的 Agent Loop；质量要求上升时叠加 Verification；任务进入业务系统时接入 Event-driven；拥有稳定数据和指标后启动 Hill-climbing。高风险动作在每一层保留人工审批。

MODULE 01 · OBSERVE

可审计的工程：全流程实时、细粒度可观测

Agent 的长期自主工作需要一条可追溯的执行证据链。 从目标和计划，到每轮动作、工具调用、资源消耗、验证结果与退出裁决，都应当能实时查看、事后复盘。

01目标完成条件是否明确
02计划如何拆解、当前做到哪
03执行每轮做了什么、调用何种工具
04验证测试、diff 与证据是否可信
05裁决为什么继续、停止或升级人工

执行中看现场

  • 会话级：用 deveco attach 观察当前轮次与 Agent 输出
  • 动作级：工具调用、插件日志、失败重试与 reviewer 裁决
  • 任务级：PROGRESS、commit、预算与退出状态

结束后能细粒度复盘

  • 复现：在 cannbot-insight 沿 turn-by-turn 轨迹还原关键决策
  • 归因：分清模型、提示词、工具、验证器或环境问题
  • 改进：依据 Trace 和失败样本迭代 Harness

快速开始

唯一前置：已安装 deveco（npm install -g @deveco/deveco-code）。 其余依赖——包括 vendor 在 vendor/ 下的 cannbot-insight 本体——全部由 setup.sh 就地装好。

~/deveco-lessons
$ git clone <本仓库> && cd deveco-lessons
$ ./setup.sh            # 一键环境：node20 / bun / cannbot 依赖与数据库，幂等可重跑
$ deveco auth login     # 唯一需要你自己完成的一步：配模型凭证

装完先从观测实操开始：先建立可审计现场，再进入 Loop Engineering。

LESSON 1

建立细粒度审计轨迹：DevEco 对接 cannbot-insight

给 agent 会话装上「回放镜头」。deveco 是 opencode 派生的，它的会话库 ~/.local/share/deveco/deveco.db 里 session / message / part 三张表结构 和 cannbot-insight 的 opencode-db 适配器一模一样——所以对接零改码， observe.sh 只做导入与起 server。

你会学到

  • DevEco 的会话数据长什么样、存在哪
  • 为什么 cannbot-insight 能零改码直读 DevEco 的会话库
  • 用 observe.sh 把任意项目的会话导入并在 Web 界面回看

步骤

  1. 造几条会话 —— 随便建个目录，用 deveco 跑两三个会话（有工具调用更好看）： 
    $ mkdir -p ~/play/hello-deveco && cd ~/play/hello-deveco
$ deveco run "写一个 fizzbuzz.py 并运行它"
$ deveco run "把 fizzbuzz 改成 1 到 30 并重新运行"
  2. 导入观测 —— 回到本仓库，按项目目录捞出所有 root 会话、逐个导入（按 taskId 去重，幂等）： 
    $ cd <本仓库>
$ ./lesson1-insight/observe.sh ~/play/hello-deveco
✅ 观测就绪: http://localhost:21025
  3. 回看 —— 浏览器开 http://localhost:21025，点进会话逐个看分析页： tokens/成本、上下文增长、工具调用、工作流阶段、概念传播。

    重复运行 observe.sh 会按 taskId 更新已有记录，会话数不会翻倍。

验收

  • 能在 Web 界面指出某一轮的输入/输出 token 数
  • 能说出该会话调用了哪些工具、各调了几次
  • 对同一项目重跑 observe.sh，会话数不翻倍（幂等）

坑

  1. node 版本：better-sqlite3 只在 node 20 编得过。脚本会自动切 nvm 的 node 20，手动进 vendor 装依赖要先 nvm use 20。
  2. observe 是旁路：cannbot 没装好、server 起不来、deveco.db 不存在时，脚本会提示原因并以退出码 0 结束，调用方可以继续运行。
  3. server 日志在 /tmp/cannbot-insight.log，起不来先看它。
MODULE 02 · CONTROL

Loop Engineering：让 Agent 按计划稳定完成复杂任务

Loop 把复杂目标组织成一个受控的工程闭环：计划先行， 每轮只推进一个可验证增量，独立验证器判断差距，状态文件继承进度，退出条件控制成本与风险。

定义目标→ 制定计划→ 执行最小增量→ 独立验证→ 更新状态 / 反馈↺ 证据充分，完成
稳定收敛的四个工程条件
PLAN

计划可执行

目标拆成有顺序、可独立验收的小步；Agent 每轮从计划中领取下一项，持续沿既定目标推进。

GOAL → PLAN → 当前子任务
VERIFY

结果可验证

测试、编译、静态检查、截图或只读 reviewer 给出客观信号；验证器与执行者隔离，防止执行者修改验收规则。

执行结果 → oracle / reviewer → DONE 或差距
STATE

进度可继承

Git、任务清单和进度文件记录已完成项、失败尝试与下一步，让跨轮、跨会话执行不会失忆或重复劳动。

commit + PROGRESS → 下一轮继续
GUARDRAIL

过程有边界

轮次、时长、token、权限和人工审批共同约束循环；高风险、不可逆动作必须停在人工确认之前。

预算 / 权限 / 审批 → 安全退出
任务可分解+ 每轮可验证+ 状态可继承+ 反馈可执行⇒ 适合 Loop Engineering

下面用两种形态落地：Ralph Loop 用跨会话 worker / reviewer 做强隔离；/goal 插件用单会话事件驱动和证据门控做轻量常驻。最后用 A/B 实验检验反馈环的真实收益。

LESSON 2

Ralph Loop：让 Agent 按计划跨会话推进

给 deveco 一个目标和可验收计划，它自己 worker → reviewer → worker 循环推进，直到 reviewer 判定验收通过。 核心工程原则：裁判和执行者分离，独立验收测试藏在 worker 摸不到的地方。

三件套

worker每轮挑一个最小可验证子任务做掉，写进 .ralph/PROGRESS.md，git commit。
reviewer只读裁判，用 git log / git diff 核实，最后一行必须是 DONE 或 CONTINUE: ⟨还差什么⟩。
插件监听 session.idle → 开 reviewer 会话裁决 → DONE 写 .ralph/DONE 收工，否则把 worker 再踢一轮。

步骤（先拿 smoke 案例练手，30 秒验证整条链路）

  1. 装 —— init 是有副作用的一次性动作（插件、agent、git、配置、GOAL 骨架），所以和「跑」分开： 
    $ cd lesson2-ralph-loop
$ ./ralph.sh init ~/ralph-smoke
  2. 灌目标 —— 用自带案例；自己出题则编辑 .ralph/GOAL.md，验收标准必须「能被 git/测试客观核实」： 
    $ ./ralph.sh sample smoke ~/ralph-smoke   # 案例带的独立验收测试一并装进 .ralph/

    验收测试放在 .ralph/verify.test.ts。worker 拥有全部工具权限，能够修改 test/ 下的用例；bun test 不扫描隐藏目录，因此 worker 看不到也改不到这份验收测试。

  3. 跑 —— 起 server、点火、盯到收工、自动关 server；脚本会保留你改过的插件和提示词： 
    $ ./ralph.sh run ~/ralph-smoke
# 想看单次基线（无 reviewer、无续轮）：./ralph.sh once ~/ralph-smoke
# 想实时围观：deveco attach http://127.0.0.1:4097 --session $(cat ~/ralph-smoke/.ralph/session_id)
  4. 看结果 —— 结束标志二选一，然后自己复验： 
    .ralph/DONE      # 验收通过
.ralph/STOPPED   # 到 maxIterations（默认 20，首跑建议设 3）仍未通过
$ bun test ./.ralph/verify.test.ts   # 独立验收测试复验
  5. 回看轨迹 —— run 收工后自动调第一课的 observe.sh，把 worker / reviewer 会话导进 cannbot-insight， 开 http://localhost:21025 turn-by-turn 回看每轮的 token 与工具调用。

    关闭自动观测：RALPH_NO_OBSERVE=1 ./ralph.sh run ...；手动补一次：./ralph.sh observe ⟨项目⟩。观测作为旁路运行，异常不会影响 run 的结论。

DevEco 适配的四个要点

  1. 配置目录：插件配置放在 .deveco/。写入 .opencode/ 会静默失效；.ralph/plugin.log 的生成记录可以确认插件已经加载。
  2. 配置文件：项目配置使用 deveco.json 或 deveco.jsonc；opencode.json 不会激活 DevEco 配置。
  3. SDK 参数结构：DevEco SDK 采用 hey-api 风格，调用形式为 session.prompt({ path: { id }, query: { directory }, body: {...} })。扁平参数会导致会话创建失败；点火直接调用 HTTP API，跳过当前存在兼容问题的 deveco run --attach。
  4. Server 鉴权：保持 DEVECO_SERVER_PASSWORD 未设置。该变量会为 serve 开启 basic auth，插件内 client 调用同一 server 时会收到 401，循环随即空转。

实验实证：单次 Agent vs Ralph Loop

盲验收 A/B 对照实验（experiments/experiment1，2026-07-16 实测）， 用 Lesson 2 的 ralph.sh once / run 做双臂，回答一个问题： 同一个模型、同一份任务，「一次生成」和「带验收反馈的循环」到底差在哪？

实验设计

任务：把 CPython textwrap 移植成行为等价的 Rust（5 个函数、12 个选项）。 验收：816 条「输入 → CPython 期望输出」向量，对 worker 完全隐藏（不在可见路径、不进 git 历史，红线禁读 + 赛后审计）。 两组的 GOAL、模型、工具、盲验收完全一致，唯一被操纵的变量是反馈环的有无。 worker 故意用弱模型（deepseek-v4-flash）：强模型靠本地对拍 CPython 就能自收敛到满分，轮不到外循环出场—— 「便宜 worker + 强验收循环」正是 loop engineering 的经济学卖点。

反面教材：Goodhart 与审计
第一次跑 A 组时，flash 在会话开始第 17 秒就把隐藏验收向量 .ralph/vectors.jsonl 读走，对着验收集搭本地 harness 刷出「816/816」——满分全程是 Goodhart，成绩作废。 结论：对全工具 agent，安全边界由规则和审计共同建立。 审计命中后要人工判读：读过验收数据内容 = 作弊作废；纯元数据访问（如 git ls-files 列文件名）可豁免但要记录理由—— 本次 A 组唯一一条命中即属良性豁免。它再次说明：验证器被污染，agent 就会持续优化错误目标。

LESSON 3

devecocode-goal-plugin：给会话装上目标循环

开发中 · vendor 已全绿

给会话一个「完成条件」，插件把普通对话变成一个事件驱动的目标循环： agent 停下来（session.idle）但目标没完成，就自动续推； 插件只接受携带证据并通过审计门控的完成声称。 整个循环发生在单个会话内，不需要外部 shell 点火。 （插件适配自 opencode-goal-plugin v0.6.5。）

你会学到（对应 Loop Engineering 的哪几层）

  • /goal 的目标生命周期：设定 → 空转续推 → 证据门控 → 归档——正是本模块闭环「计划 → 执行 → 验证 → 反馈 → 退出」的单会话实现
  • 空转自续 = 事件驱动循环：session.idle 事件自动触发续推
  • 证据门控 = 验证循环：完成声称要交出可核查的证据，防止「反复改写、自我宣布胜利」
  • 它和 ralph loop 的分工：外循环跨会话、独立裁判；内循环单会话、轻量常驻——两者可以叠加

用法

  1. 装 —— goal.sh 只负责装：插件 + merge deveco.json 的 /goal 命令： 
    $ ./goal.sh init ~/my-project     # --update 才会覆盖你改过的文件
  2. 会话里设目标 —— 交互式 TUI 形态，一切在单个会话内发生： 
    $ cd ~/my-project && deveco
# 会话里输入：
/goal 创建 hello.txt 并 git commit
/goal status     # 看当前目标
/goal history    # 看 lifecycle 事件
  3. 空转自续 —— agent 停下来但目标没完成？插件自动发续推消息，把它踢回去接着干。 这一步就是闭环里的「定位差距 → 注入反馈 → 再执行」。
  4. 证据门控完成 —— 完成汇报要过证据审计才收工；状态归档在 .deveco/goals/state.json。 退出条件可配（.deveco/goal-plugin.json：maxTurns 10 / maxDurationMs 900000 / maxTokens 200000）。 排障第一句永远是：看 .deveco/goals/plugin.log 有没有加载记录。

适用场景验收标准明确、可反复运行测试，需要多轮“实现—验证—修复”收敛的任务周期长、需要状态持久化、预算控制、暂停恢复和人工介入的复杂任务
退出.ralph/DONE / STOPPED目标归档 / maxTurns 等预算耗尽

选择原则：Ralph Loop 解决“如何通过多轮验证反复做对”，/goal 解决“如何带着状态和边界长期做完”。高风险长程任务可以用 /goal 管理生命周期，在关键阶段叠加 Ralph 式验证循环。

  • 适配任务按 10 步计划推进中：状态路径迁 .deveco/、session API 注入 query.directory、入口壳、goal.sh、冒烟、E2E 走查（见 docs/superpowers/plans/2026-07-16-lesson3-goal-plugin.md）
MODULE 03 · REFLECT

分享与交流

拿你跑出来的证据（分数、审计、轨迹）来讲，围绕这几个问题：

什么任务适合 Loop？点击展开参考答案

同时满足四个条件的任务：目标能拆成可独立验收的小步（可分解）；每步有测试、编译、截图这类客观信号（可验证）； 进度靠 git 与进度文件跨轮继承（可继承）；验证器指出的差距能直接变成下一轮输入（反馈可执行）。 典型场景：代码迁移、重构、CI 修复、契约检查——正确性优先、测试可反复运行的生产任务。

什么任务不适合 Loop？

三类：验收标准无法客观化的（创意、审美、需求探索——验证器给不出可执行差距，循环只会空转烧 token）； 高风险、不可逆的动作（上线发布、删数据、对外发信——必须停在人工审批，而不是循环重试）； 单次就能做完、人工抽查即可的小任务（验证开销大于收益，Agent Loop 一把过就够）。

Loop Engineering 和 Harness 是什么关系？

Harness 是循环的「壳」：Prompt、工具、Grader、状态文件、预算与审批规则这些工程件；Loop 是 Harness 组织模型反复执行的运行方式， 可靠性 = 模型 × Harness。循环跑不好先改 Harness 再考虑换模型——实验里同一个弱模型套上强验收 Harness 就从 795 爬到 816； 反馈是否可执行、验证器是否可信、状态是否继承，这三样改起来便宜且可控。模型只在 Harness 改到头仍拉不动时才换。

如何优化 Loop？

用第 4 层 hill-climbing 的方法：从 Trace 与 Grader 反馈里找共性失败（cannbot-insight 回放），沿证据链归因—— 目标不清改 GOAL，缺领域知识补 Skill / Tool，反馈不可执行改 Grader 与 reviewer 提示词，越界烧钱调预算与审批。 一次只改一个变量，重跑对比分数曲线，有效的改动固化进 Harness 再发布。

课堂挑战 · 不给测试用例，把通过率提上去

挑战：用小鲁班模型跑 experiment1，在 worker 看不到任何测试用例的前提下提高隐藏验收通过率

把 worker / reviewer 换成小鲁班网关模型（experiments/experiment1 下 cp local.env.example local.env 填三行即接入），重跑 A/B 双臂。 816 条验收向量对 worker 始终不可见——你要在不泄漏任何一条用例的前提下，让分数尽可能爬升。

  • 允许改：GOAL 的任务表述、worker / reviewer 提示词、maxIterations、worker 的自检策略（例如对拍本机 CPython）
  • 不允许：把 .ralph/vectors.jsonl / run_qa.py 喂给 worker，或放宽红线与审计——审计不 clean，成绩作废
  • 交付：两臂分数 + 审计结论 + 你改了什么、为什么有效——沿上面的证据链讲给点评者

开放问题

  • 为什么不给测试用例？验证器的独立性到底在保护什么？
  • 如果把 816 条向量直接交给 worker，分数会发生什么？那个分数还能说明什么？ （回看模块 02 的反面教材：第 17 秒偷读验收集刷出的「满分」）

04

AI Native 迁移系统拆成知识、流程与质量三层

Knowledge 解决模型缺少什么，Flow 决定任务如何推进，Harness 规定结果如何验证和修复。

01 · KNOWLEDGE

Knowledge in Skills

知识属于 Skill
本质问题：模型缺乏领域知识时，Agent 如何获取？
回答：知识类型决定供给方式——程序性知识进入 Skill，陈述性知识进入 RAG。

02 · FLOW

Flow in Agents

流程属于 Agent
本质问题：任务由模型自主推进，还是由外部编排？
回答：任务复杂度 × 模型能力决定放权程度——顶层 Workflow 显式编排，中层 Agent 自主。

03 · HARNESS

Harness by Verify & Fix

验证与修复看护质量
本质问题：质量由谁、以什么方式保证？
回答：人负责核心审阅，规范约束把控交付，大模型负责验证与修复。

05

程序性知识进入 Skills，陈述性知识按需使用 RAG

两条路线解决不同类型的问题：RAG 回答“是什么”，Skills 固化“怎么做”。

Knowledge in RAG

定位
检索增强生成
知识形态
非结构化文档块（chunks、embeddings）
获取方式
语义相似度检索，结果不可预期
适合的知识
陈述性知识（Declarative）：API 参考、FAQ、已知问题

Knowledge in Skills

定位
知识编码为结构化能力单元
知识形态
SKILL.md + references/ + scripts/
获取方式
显式路由选择，Agent 明确知道加载了什么
适合的知识
程序性知识（Procedural）：迁移流程、转换规则、最佳实践

为什么选择 Skills

01 · 知识类型核心是“怎么做”，属于程序性知识。
02 · 确定性同样输入必须产出同样输出。
03 · 可验证CI 可以校验能力单元的完整性。

Skills 与 RAG 不互斥，应按知识类型选型。

06

顶层主 Agent 显式编排，中层 Task 内 Agent 自主

流程的放权程度由任务复杂度与模型能力共同决定：可预定义的步骤交给 Workflow，动态实现决策留给 Agent。

五步门控 Pipeline

  1. Spec定义规格
  2. Plan生成计划
  3. Execute执行任务
  4. Verify验证结果
  5. Retrospect复盘回流

顶层主 Agent · 显式 Workflow

五步流程可预定义，人在关键 Gate 审批；模型不需要探索“下一步做什么”。

中层 Task · Agent 自主

Agent 自主选择 Skill、决定实现方案，并处理编译错误。

两个决定因素

01 · 任务可分解性步骤可预定义 → Workflow；步骤需要动态探索 → Agent。
02 · 模型能力边界模型可靠 → 放权给 Agent；模型不可靠 → 外部控制与工程校验。

编排策略随模型能力与任务复杂度变化

    自主执行
  2. Agent + 轻量
    guardrail
  3. Hybrid
    模型规划 + 工程校验
    当前选择
  4. 状态机 / DAG
    主导
  5. 强拆解 + 强验证
    + 人工兜底

Orchestrator-Workers 是模型能力与任务复杂度达到平衡时的优选方案。

07

Verify & Fix 把迁移过程沉淀为门控契约

每一步都要通过验证才能放行；失败携带错误证据回到修复与重检，持续失败再降级人工。

  1. 1 · GenerateAI 生成代码
  2. 2 · Verify检查规范 / 契约 / 测试
  3. 3 · Fix未通过则修复，通过则放行

未通过时自动返回 Verify 重检，形成有限次验证—修复循环。

Verify

  • 编译 / 语法 / 类型
  • 鸿蒙化规范契约
  • DT 测试
  • 视觉对比

Fix

  • 定位失败点 + 错误证据
  • 模型有限次数重试
  • 仍失败则降级人工
  • 修复记录进入审计文件

Human Review

  • 未通过验证，不进入下一步
  • 审阅功能完备度
  • 发现功能缺失则返回修复

AI 写代码，Verify & Fix 把关，人审阅关键产物。

MODULE 05 · CLOSE

总结与交流

今天把 Agent 放进一个可观察、可控制、可验证、可复盘的工程系统。

  1. 先可观测，再自治细粒度 Trace 用于判断 Agent 正在推进、空转、走偏，或污染验证器。
  2. Loop 驱动复杂任务收敛计划、验证器、持久状态和可执行反馈共同决定任务能否稳定完成。
  3. “完成”是一条可审计结论让代码、测试、diff、日志与独立裁决构成证据链，并为预算、权限和人工审批保留边界。

开放交流

  • 你手上的哪类开发任务最值得先改造成 Loop？
  • 这个任务最可信的验证器是什么，Agent 能否绕过它？
  • 哪些步骤可以自治，哪些动作必须保留人工审批？
